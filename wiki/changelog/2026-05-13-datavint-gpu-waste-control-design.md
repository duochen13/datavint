## Problem Statement

ML teams waste 20-30% of GPU training budget on duplicate experiments and preventable failures. Engineers running 20+ experiments lose track of what they already tried and accidentally rerun similar configurations, wasting thousands of dollars on redundant compute.

The real cost isn't $/hour—it's the massive share of training minutes spent on runs that never should have happened: duplicated experiments (20-30%), broken pipelines (30-40%), data corruption (5-15%).

**Core insight:** Every ML platform today does PASSIVE tracking (MLflow, W&B log what happened). No one does ACTIVE blocking (prevent wasteful runs before they start). DataVint sits between data pipelines and training systems as a pre-execution gate, answering: "Should this GPU run exist at all?"

## Status Quo

**Current workflow:**
- Teams don't prevent duplicates—they just let them happen
- Some manually check MLflow before launching
- Some Slack the team "has anyone run this?"
- No automated pre-execution governance

**Existing tools (all PASSIVE):**
- MLflow, W&B, Neptune: Log experiments after they run
- FinOps: Optimize infrastructure (spot instances, right-sizing), not experiments
- Approval gates: Exist for compliance (regulated industries), not cost control
- Fault tolerance: Reduce time lost to failures (92% improvement), but don't prevent wasteful runs

**The gap:** No one blocks duplicate/wasteful experiments BEFORE they consume GPU.

## Premises

1. **Duplicate experiments are 20-30% of GPU waste**
   - Confirmed by customer conversations
   - 2nd biggest source (after failed runs 30-40%)

2. **ML engineers frequently run exact duplicates by accident**
   - Engineers running 20+ experiments forget what they tried 2 weeks ago
   - No global memory of "what has already been run"

3. **Dataset hashing is cheap enough via sampling**
   - Sample 0.1% of 500GB dataset (500MB)
   - Complete fingerprinting in <30 seconds
   - Sufficient for duplicate detection

4. **Checkpoints from exact duplicates are rarely reusable**
   - Code, dependencies, data pipelines change
   - Old checkpoints often not safe to reuse
   - Goal: prevent rerun, not reuse checkpoint

5. **This is a product, not a feature (REVISED after cross-model challenge)**
   - Original premise: "20-30% GPU savings is sufficient wedge"
   - Revised: "20-30% savings + compounding value (irreplaceable database) OR catastrophic loss prevention (one $40K mistake) is sufficient wedge"
   - Rationale: Savings alone may not justify procurement. Need either (1) database becomes irreplaceable or (2) prevents one disaster
   - Evidence that would prove this wrong: Free tier users love it but conversion stalls at procurement; teams say "we'll build this internally"

6. **"Experiment governance" vs "experiment tracking" is the moat**
   - MLflow/W&B are passive trackers
   - DataVint is active pre-execution gate
   - Enforces policies, not just logs

## Approaches Considered

### Approach A: Lightweight CLI Gate ⭐ **RECOMMENDED**

**Summary:** CLI tool that users run before launching training jobs. Queries a central fingerprint database, blocks exact duplicates, warns on near-duplicates.

**Effort:** S (1 engineer, 6-8 weeks)

**Risk:** Low (simple architecture, no platform dependencies)

**Pros:**
- Ships fastest—minimal moving parts, proven pattern (like pre-commit hooks)
- Platform-agnostic—works with SageMaker, Vertex, Databricks, local training
- Low adoption risk—it's informational first, users control final decision
- Reuses existing infrastructure—just needs postgres + simple API
- Bottom-up adoption—one engineer tries it, tells teammates

**Cons:**
- Requires manual adoption—users must remember to run `datavint check` before every job
- Easy to skip—no enforcement, relies on discipline
- Limited data quality checks—only fast sampling-based checks (<30s)
- No automatic blocking—warns but can't prevent launches without user cooperation

**Reuses:** Existing data profiling code, dataset hashing logic

### Approach B: Platform Integration Layer

**Summary:** API that hooks into MLflow/SageMaker/Vertex launch workflows. Every training job automatically checks for duplicates before execution.

**Effort:** M (1-2 engineers, 8-12 weeks)

**Risk:** Medium (platform API dependencies)

**Pros:**
- Automatic enforcement—no manual CLI step
- Team-wide adoption—once configured, whole team covered
- Richer data quality checks—more time for analysis
- Audit trail—every blocked/warned run logged

**Cons:**
- Platform-dependent—separate integrations for each platform
- Higher adoption friction—requires IT/platform team buy-in
- Longer development time—8-12 weeks
- Breaking changes risk—platform API updates could break integrations

**Reuses:** Data profiling, fingerprinting, adds platform layer

### Approach C: Rec Systems Vertical

**Summary:** Pre-execution gate built SPECIFICALLY for recommendation systems. Understands rec-specific waste patterns (O2O issues, behavioral feature duplication).

**Effort:** M (2 engineers, 8-10 weeks)

**Risk:** Medium (narrow market)

**Pros:**
- Domain expertise advantage—deep Meta rec systems knowledge
- Higher value per customer—rec-specific checks
- Premium pricing justified—vertical expertise
- Clearer positioning

**Cons:**
- Extremely narrow market—excludes CV, NLP, robotics
- Expansion requires rework—can't generalize rec logic
- Competed with original design doc—already pivoted away once
- Might not match committed customer—they wanted general ML solution

**Reuses:** Rec systems knowledge, data quality detectors

## Recommended Approach

**Approach A: Lightweight CLI Gate**

**Rationale:**

1. **Fastest to ship** - 6-8 weeks, delivers on timeline
2. **Lowest adoption risk** - CLI is familiar, no platform team buy-in required
3. **Validates core value prop** - proves "experiment governance prevents waste" without complex integrations
4. **Platform for expansion** - once users see value, they'll ASK for platform integrations (Approach B)
5. **Incorporates cross-model insight** - local-first approach tests UX before infrastructure investment

Start with the wedge that proves the insight without betting on complex integrations. If engineers start relying on `datavint history` to remember what they tried, that's compounding value. If it catches one $4000 duplicate in the first month, that's instant ROI.

## Open Questions

1. **Adoption mechanics:** How do we get from "first individual user" to "whole team using it"? Viral adoption via Slack screenshots? Team installation script?

2. **Database sync:** CLI talks to central database—how do we handle teams that can't send dataset hashes to external service? Local-only mode?

3. **Near-duplicate threshold:** Exact match is clear. But 98% similar? 95%? What's the threshold that feels helpful vs noisy?

4. **Cost estimation:** Should we show "you already ran this (estimated $3200)" or just "you already ran this"? Requires user to configure GPU pricing.

5. **Expansion timing:** When to add platform integrations (Approach B)? After X users? After $Y ARR? After customer requests?
